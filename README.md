# News React

## 1. Краткое описание

News React — учебный новостной интерфейс на React, который получает данные из внешнего News API и показывает последние новости, фильтрацию по категориям, поиск и пагинацию. Проект сделан как практическая отработка архитектуры frontend-приложения: разделение по слоям, типизация контрактов данных, работа с асинхронным состоянием и повторное использование UI-компонентов.

## 2. Цели проекта

Основная цель проекта — не собрать витрину новостей, а отработать инженерные решения, которые встречаются в прикладных React-приложениях:

- построение структуры по Feature-Sliced Design с явными границами между `app`, `pages`, `widgets`, `entities` и `shared`;
- разделение серверного состояния, пользовательских фильтров и презентационного UI;
- работа с внешним API через типизированный слой запросов;
- минимизация лишних запросов при поиске через debounce;
- переиспользование общих UI-паттернов: skeleton-состояния, пагинация, слайдер категорий, обертка для изображений;
- настройка базовых инструментов контроля качества: ESLint, TypeScript, Steiger.

## 4. Архитектура и структура

Проект организован по принципам Feature-Sliced Design. Верхний уровень `app` отвечает за инициализацию приложения, подключение провайдеров, store и базовый layout. `pages` собирает сценарии конкретной страницы из сущностей и shared-компонентов. `entities/news` содержит доменную сущность новости: API, slice и UI-компоненты для отображения новостных элементов. `shared` хранит переиспользуемые инфраструктурные и UI-части, которые не завязаны на конкретную фичу.

Такая структура выбрана, потому что даже для небольшого проекта она фиксирует направление зависимостей и не дает бизнес-логике, API-запросам и базовым компонентам смешиваться в одном слое. Компоненты страницы знают о сценарии использования, сущность новости знает о своем API и отображении, а shared-слой остается максимально нейтральным.

Контроль архитектурных правил вынесен в Steiger. Это важно, потому что FSD легко нарушить незаметно: импорт из неправильного слоя или протечка доменной логики в shared обычно не ломают сборку сразу, но ухудшают сопровождаемость.

## 5. Ключевые решения

### RTK Query как граница серверного состояния

Получение новостей, последних новостей и категорий реализовано через `createApi`. Это отделяет серверное состояние от локального UI-состояния и убирает ручное управление состояниями загрузки, кешированием и жизненным циклом запроса.

В результате компоненты страницы работают с декларативными hooks (`useGetNewsQuery`, `useGetLatestNewsQuery`, `useGetCategoriesQuery`) и не содержат собственного кода для выполнения HTTP-запросов.

### Фильтры вынесены в Redux-состояние

Параметры фильтрации хранятся в `newsSlice`: номер страницы, размер страницы, категория и ключевые слова. Это делает фильтры единым источником правды для списка новостей, пагинации и поисковой строки.

Такой подход оправдан, потому что несколько независимых компонентов участвуют в одном сценарии: категории меняют фильтр, поиск меняет keywords, пагинация меняет pageNumber, а запрос новостей строится из общего состояния.

### Debounce перед запросом поиска

Поисковая строка обновляет Redux-состояние сразу, но запрос использует debounced-значение. Это снижает количество обращений к API при вводе текста и оставляет интерфейс отзывчивым.

Компромисс здесь в том, что состояние поля и состояние запроса расходятся на время задержки. Для текущего сценария это приемлемо: пользователь видит введенный текст сразу, а сеть нагружается только после паузы.

### Skeleton через HOC

Состояния загрузки вынесены в HOC `withSkeleton`, который оборачивает списки новостей и баннеров. Это позволяет не дублировать условную отрисовку loader-состояний внутри каждого компонента списка.

Решение подходит для проекта, где loading-состояния повторяются, но отличаются типом skeleton, количеством элементов и направлением. При этом сами списки остаются сфокусированы на отображении данных.

### Унификация презентационных компонентов

Общие элементы вроде `Pagination`, `PaginationWrapper`, `Search`, `Slider`, `Image` и `Skeleton` лежат в shared-слое. Это снижает связность page-слоя и позволяет строить экран из небольших компонентов с понятной ответственностью.

Например, `PaginationWrapper` решает задачу размещения пагинации сверху и снизу без дублирования разметки в странице, а `Image` централизует обработку отсутствующих изображений.

### Тема через Context

Светлая и темная тема реализованы через `ThemeProvider` и `data-theme` на корневом контейнере приложения. Это простое решение без глобального state manager, потому что тема не участвует в доменной логике и не требует синхронизации с API.

## 6. Интересные моменты реализации

- API-ключ и базовый URL вынесены в переменные окружения `VITE_NEWS_API_KEY` и `VITE_NEWS_BASE_API_URL`, поэтому клиентский код не привязан к конкретному endpoint в исходниках.
- Категории загружаются отдельным запросом и отображаются в горизонтальном слайдере, что отделяет справочные данные от основного списка новостей.
- Типы API-ответов и доменных структур описаны явно: новости, категории, фильтры, параметры запросов и props пагинации имеют отдельные TypeScript-контракты.
- UI строится через CSS Modules, поэтому стили компонентов локализованы и не зависят от глобальных селекторов, кроме базовой темы и контейнера приложения.
- В проекте есть архитектурная проверка `npm run lint:fsd`, что полезно именно для поддержания структуры, а не только для синтаксической чистоты.

## 7. Что можно улучшить

- Сейчас `TOTAL_PAGES` задан константой. Лучше получать количество страниц или общее число результатов из API, если endpoint это поддерживает.
- `setFilters` принимает произвольный `key: string`, из-за чего часть типобезопасности теряется. Более надежный вариант — сделать отдельные actions для каждого фильтра или типизировать ключ через `keyof IFilters`.
- При смене категории или поискового запроса стоит сбрасывать `pageNumber` на первую страницу, чтобы пользователь не оставался на старой странице в новом наборе результатов.
- Ошибки API пока не имеют отдельного UI-состояния. Для production-сценария нужны empty/error states и различение "нет данных" от "запрос не выполнился".
- Тема хранится только в runtime-состоянии. Если темная тема должна быть пользовательской настройкой, ее стоит сохранять в `localStorage` или системно читать из `prefers-color-scheme`.
- ESLint сейчас настроен только для `js/jsx`, хотя проект написан на TypeScript. Имеет смысл добавить type-aware правила для `ts/tsx`.

## 9. Запуск

Создать `.env` на основе `.env.example`:

```env
VITE_NEWS_API_KEY=
VITE_NEWS_BASE_API_URL=
```

Установить зависимости:

```bash
npm install
```

Запустить dev-сервер:

```bash
npm run dev
```

Собрать production-версию:

```bash
npm run build
```

Проверить lint:

```bash
npm run lint
```

Проверить FSD-правила:

```bash
npm run lint:fsd
```
